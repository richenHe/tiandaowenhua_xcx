## 一、测试概念入门

### 1.1 什么是接口测试？

**接口测试（API Testing）** 是验证系统接口是否按照设计正确工作的测试方法。

#### 📌 与单元测试的区别

| 对比项 | 单元测试 | 接口测试（当前使用） |
|--------|---------|---------------------|
| **测试对象** | 单个函数/方法 | 完整的 HTTP 接口 |
| **测试范围** | 代码的最小单元 | 从请求到响应的完整流程 |
| **依赖环境** | 隔离（使用 Mock） | 真实环境（数据库、云函数） |
| **执行速度** | 极快（毫秒级） | 较慢（秒级） |
| **测试价值** | 验证代码逻辑 | 验证业务功能 |

**示例对比**：

```javascript
// ❌ 单元测试例子（不是我们使用的）
test('计算总价', () => {
  expect(calculateTotal(100, 0.9)).toBe(90);
});

// ✅ 接口测试例子（我们使用的）
{
  name: '获取订单列表',
  method: 'GET',
  url: '/api/admin/order/getOrderList',
  expectedFields: ['total', 'list']  // 验证真实接口返回
}
```

### 1.2 测试系统类型定位

**本测试系统 = 接口测试 + 集成测试**

- ✅ 测试完整的 API 端到端流程
- ✅ 使用真实的云开发环境
- ✅ 验证数据库操作是否正确
- ✅ 检查业务逻辑完整性

## 三、测试标准（判定规则）

### 3.1 四层验证机制

每个接口测试会经过 **4 个层次** 的验证，任何一层失败则测试失败：

#### ✅ Level 1：HTTP 状态码检查

```javascript
// 必须返回 200 OK
if (!response.ok) {
  ❌ 测试失败
}
```

**失败原因**：
- 404 Not Found - 接口地址错误
- 500 Internal Server Error - 服务器内部错误
- 网络连接失败

---

#### ✅ Level 2：业务成功标识

```javascript
// 返回数据必须包含 success: true
if (!result.success) {
  ❌ 测试失败
}
```

**失败原因**：
- `code: 401` - Token 过期或无效
- `code: 400` - 参数错误
- 业务逻辑错误（数据不存在、权限不足等）

---

#### ✅ Level 3：必需字段完整性

```javascript
// 检查预定义的必需字段
expectedFields: ['total', 'list']

// 所有字段必须存在
const allFieldsExist = expectedFields.every(field => 
  response.data[field] !== undefined
);

if (!allFieldsExist) {
  ❌ 测试失败: "缺少必需字段: total"
}
```

**测试用例示例**：

```javascript
{
  name: '获取课程列表',
  expectedFields: ['total', 'list'],        // 必须有这 2 个字段
  itemFields: ['id', 'name', 'type']        // list 中每项必须有这 3 个字段
}
```

---

#### ✅ Level 4：数据类型验证

```javascript
// 验证字段类型
fieldChecks: [
  {
    field: 'total',
    exists: true,
    type: 'number'  // ✅ 类型正确
  }
]
```

---

### 3.2 测试通过标准

**所有 4 层验证都通过 = ✅ 测试成功**

```
HTTP 200 ✅
  ↓
success: true ✅
  ↓
必需字段完整 ✅
  ↓
数据类型正确 ✅
  ↓
🎉 测试通过
```

## 五、测试数据管理

### 5.1 数据添加规范 ⭐

遵循 **优先级原则**：

#### 🥇 优先：通过接口创建

```javascript
// ✅ 好的做法：测试创建接口的同时添加数据
{
  name: '创建课程',
  method: 'POST',
  body: { 
    name: '测试课程',
    type: 1,
    currentPrice: 1688   // ✅ 用驼峰命名（前端→云函数规范）
  }
}
```

**适用场景**：
- ✅ 课程、排期、案例、资料
- ✅ 大使活动、公告、配置
- ✅ 所有有"创建接口"的数据

---

#### 🥈 特殊情况：直接插入数据库

```sql
-- 仅在以下情况使用 SQL 插入
INSERT INTO tiandao_culture.appointments (user_id, course_id, ...)
VALUES (30, 1, ...);
```

**适用场景**：
- 无管理端创建接口（如 `appointments`）
- 依赖复杂业务逻辑（如 `user_courses` 需要支付流程）
- 关联关系数据（预约需要关联用户课程）

---

### 5.2 测试数据标记规范

**所有测试数据必须明确标识**：

```javascript
// ✅ 正确示例
{
  name: '测试课程',           // 名称以"测试"开头
  order_no: 'TEST20260215001', // 订单号带 TEST 前缀
  user_id: 30,                // 使用固定测试用户
  real_name: '测试用户'
}

// ❌ 错误示例
{
  name: '天道文化课程',        // 看起来像真实数据
  user_id: 50                 // 使用随机用户
}
```

**标记规则**：
1. 名称字段：`测试XXX` 或 `Test XXX`
2. 订单号：`TEST` 前缀
3. 用户 ID：固定使用 `30`（测试专用用户）
4. 状态：可以是任何测试需要的状态

---

### 5.3 数据清理策略

**❌ 不需要清理测试数据**

理由：
- 保留数据用于下次测试
- 避免重复创建浪费时间
- 测试数据有明确标识，不会混淆

**例外情况**：
- 测试删除接口时，允许删除测试数据
- 数据库空间不足时，可手动清理

---

### 5.4 状态型测试数据——每次测试前必须重置 ⭐

部分测试用例会**消耗数据状态**（如审核、签到、退款），下次运行时若状态已变，测试会失败。

#### ⚠️ 必须在每次测试前重置的数据

| 测试用例 | 表 | 条件 | 需要的状态 |
|---------|---|------|----------|
| 提现审核 | `withdrawals` | `id=3` | `status=0`（待审核）|
| 审核大使申请 | `ambassador_applications` | `id=2` | `status=0`（待审核）|
| 审核大使活动 | `ambassador_activity_records` | `id=2` | `status=0, merit_points=0`（待审核）|
| 批量签到 | `appointments` | `id=3` 和 `id=4` | `status=0`（待签到）|

> **注意**："更新预约状态"测试会把 `id=4` 改为 `status=2`，因此批量签到需要 `id=3` 也是 `status=0`。

#### 重置 SQL（每次测试前执行）

```sql
-- 提现审核：重置为待审核
UPDATE tiandao_culture.withdrawals 
SET status=0, audit_time=NULL, audit_admin_id=NULL, audit_remark=NULL 
WHERE id=3;

-- 大使申请：重置为待审核
UPDATE tiandao_culture.ambassador_applications 
SET status=0, audit_time=NULL, audit_admin_id=NULL 
WHERE id=2;

-- 大使活动：重置为待审核
UPDATE tiandao_culture.ambassador_activity_records 
SET status=0, merit_points=0, audit_time=NULL, audit_admin_id=NULL, audit_remark=NULL 
WHERE id=2;

-- 预约记录：重置为待签到（两条都需要）
UPDATE tiandao_culture.appointments 
SET status=0, checkin_time=NULL, checkin_admin_id=NULL 
WHERE id IN (3, 4);
```

---

### 5.5 删除型测试数据——每次测试前必须确认存在

测试"删除"类接口时，需要确保**目标数据存在**，且**创建数据的前置测试不会冲突**。

#### 删除型测试数据管理

| 测试用例 | 表 | 需要确保的数据 |
|---------|---|-------------|
| 创建管理员（test 4） | `admin_users` | `username='test_admin'` **不存在**（让创建成功）|
| 删除管理员（test 6） | `admin_users` | `id=11, username='test_delete_admin'` **存在** |
| 创建公告（test 12） | `announcements` | 无特殊要求，每次创建新记录 |
| 删除公告（test 14） | `announcements` | `id=20` **存在** |
| 删除反馈（test 21） | `feedbacks` | `id=3` **存在** |

#### 管理 SQL（每次测试前执行）

```sql
-- 删除上次测试创建的 test_admin（让"创建管理员"可以重新创建）
DELETE FROM tiandao_culture.admin_users WHERE username='test_admin';

-- 确保 test_delete_admin 存在（供"删除管理员"使用）
-- 若 id=11 不存在，重新插入（一般不需要，除非上次删除测试成功了）
INSERT IGNORE INTO tiandao_culture.admin_users 
  (id, username, password, real_name, role, status) 
VALUES 
  (11, 'test_delete_admin', MD5('delete123'), '待删除管理员', 'editor', 1);
```

> **循环问题说明**：删除测试本身会消耗数据（每次运行删掉一条），因此在下次测试前需要重新准备。建议将上述重置 SQL 整理成脚本，每次测试前执行一次。

## 七、常见问题处理

### 7.1 Token 过期

**现象**：
```
❌ 登录已过期
❌ code: 401
```

**解决方案**：
```
1. 测试工具会自动重新登录
2. 如果自动登录失败，点击 [手动登录] 按钮
3. 检查管理员账号密码是否正确
```

---

### 7.2 数据库连接错误

**现象**：
```
❌ "Unknown database 'cloud1-0gnn3mn17b581124'"
```

**原因**：使用了错误的数据库名

**解决方案**：
```sql
-- ✅ 正确：使用业务数据库
SELECT * FROM tiandao_culture.users;

-- ❌ 错误：使用环境 ID 作为数据库名
SELECT * FROM cloud1-0gnn3mn17b581124.users;
```

---

### 7.3 接口返回格式错误

**现象**：
```json
{
  "body": "{\"success\":true,...}"  // 字符串格式
}
```

**原因**：CloudBase HTTP Access Service 返回格式

**解决方案**：测试工具会自动解析 `body` 字段

```javascript
// 自动处理
if (result.body && typeof result.body === 'string') {
  result = JSON.parse(result.body);
}
```

---

### 7.4 字段名不一致

**现象**：
```
❌ "缺少必需字段: course_id"
```

**原因**：接口返回 `id`，测试期望 `course_id`

**解决方案 A**：修改接口返回
```javascript
// ✅ 返回明确的字段名
return response.success({ course_id: id }, '创建成功');
```

### 7.7 参数命名规范：驼峰 vs 下划线

**本项目的命名规范**（来自 `.cursorrules`）：

```
前端 → 云函数：驼峰命名（camelCase）
云函数 → 数据库：下划线命名（snake_case）
```

**对测试文件的要求**：

```javascript
// ✅ 正确：测试文件发送驼峰参数
body: {
  action: 'createCourse',
  name: '测试课程',
  currentPrice: 1688,      // ✅ camelCase
  originalPrice: 2000,     // ✅ camelCase
  courseId: 1,             // ✅ camelCase
  classDate: '2026-03-01', // ✅ camelCase
  studentName: '张三',     // ✅ camelCase
}

// ❌ 错误：测试文件发送下划线参数（云函数会提示缺少参数）
body: {
  action: 'createCourse',
  current_price: 1688,     // ❌ 应该是 currentPrice
  original_price: 2000,    // ❌ 应该是 originalPrice
  course_id: 1,            // ❌ 应该是 courseId
}
```

**云函数内部的转换职责**：

```javascript
// 云函数负责将 camelCase → snake_case 存入数据库
const currentPrice = event.currentPrice;  // 接收驼峰
await insert('courses', {
  current_price: currentPrice,            // 存入下划线
});
```

**检查参数规范的方法**：

1. 打开云函数日志，查看 `event` 对象的实际字段名
2. 对照云函数代码中 `event.xxx` 的读取方式
3. 确保测试文件发送的字段名与云函数读取的字段名一致

### 7.8 云函数路由中 `action` 字段冲突

**现象**：

```
❌ "未知的操作: list"
❌ "未知的操作: getConfigList"
```

**原因**：

测试框架通过 URL 路径传递 `action`（如 `/system?action=manageNotificationConfig`），但测试用例 body 中又发送了 `{ action: 'list' }`，导致云函数路由看到的 `action` 被覆盖为 `list`，找不到对应的 handler。

**错误示例**：

```javascript
// ❌ 错误：body 中的 action 会覆盖 URL 中的 action
{
  url: '/system',
  body: {
    action: 'manageNotificationConfig',  // URL action
    action: 'list',                      // ❌ 又传了一个 action，覆盖了上面的
    operation: 'list'
  }
}
```

**正确示例**：

```javascript
// ✅ 正确：action 只用于路由，子操作用 operation 字段
{
  name: '通知配置管理',
  url: '/system',
  body: {
    action: 'manageNotificationConfig',  // ✅ 路由 action
    operation: 'list'                    // ✅ 子操作用 operation 区分
  }
}
```

**规则总结**：

- `action` = 路由到哪个 handler（唯一）
- `operation` / `type` 等字段 = handler 内部的子操作区分
- **不要在 body 中重复传 action 字段**

### 7.9 云函数数据库字段名错误

**现象**：

```
❌ "审核活动失败"
❌ SQL error: column 'before_balance' not found
❌ SQL error: column 'change_type' not found
```

**原因**：云函数代码中写的数据库字段名与实际表结构不符

**排查步骤**：

1. 查看云函数详细日志（`getFunctionLogDetail`）
2. 找到 SQL 错误信息，定位错误字段名
3. 查看数据库详细信息文档（`docs/database/数据库详细信息.md`）
4. 对比文档中的真实字段名，修改云函数代码

**本项目已发现的字段名错误汇总**（`merit_points_records` 表）：

| 错误字段名（代码中） | 正确字段名（数据库实际）|
|--------------------|----------------------|
| `before_balance` | `balance_before` 或直接省略（非必填）|
| `after_balance` | `balance_after` |
| `change_type` | `type`（1=获得，2=消耗）|
| `description` | `remark` |
| ❌ 未传 `source` | `source`（必填，5=沙龙活动）|
| ❌ 未传 `activity_name` | `activity_name`（活动名称）|

**防止此类问题的方法**：

在写云函数 `insert()` 语句之前，先查阅 `docs/database/数据库详细信息.md` 中对应表的字段列表，确认每个字段的真实名称和是否必填。

### 9.6 测试序列的依赖关系
